# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Quick start

**Linux / macOS**
```bash
sudo apt install ffmpeg libportaudio2 rubberband-cli   # Debian/Ubuntu; use dnf/pacman equivalents elsewhere
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python main.py /path/to/music/folder
```

**Windows** (run in Windows Terminal, not cmd.exe)
```powershell
winget install Gyan.FFmpeg              # sounddevice bundles PortAudio — no separate install needed
choco install rubberband                # OR download from https://breakfastquay.com/rubberband/ and add to PATH
python -m venv .venv; .venv\Scripts\activate
pip install -r requirements.txt
python main.py C:\path\to\music
```

`main.py` checks for the `rubberband` CLI on PATH at startup and refuses to launch without it — every tempo transition is rendered through it.

The backspin transition (`B`) uses a SFX one-shot, default `samples/top_DJ_Rewind_SFX_10.mp3`, overridable via `--backspin <path>`: a bare filename is resolved inside `samples/`, a relative/absolute path is used as-is (`python main.py <music_folder> --backspin backspin_09.wav`). The resolved file must exist or the app hard-fails at startup.

On first run the app blocks at the terminal while analyzing every audio file; subsequent runs are instant because results are cached at `~/.automix_cache.json`.

## Keybindings

| Key | Action |
|---|---|
| `Enter` (on a song row) | Load as Now Playing — **only when engine is IDLE** (no track loaded, prior track finished, or after `S`). Rejected with a status message otherwise; use `N` + `M` to switch tracks during playback. |
| `N` | Queue selected song as Next Track. Replaces any queued track (discarding an in-flight Prepare); no-op if the same path is already queued. |
| `C` / `F` / `R` | Set Cue / Fade / Restore (opens inline numeric input) |
| `P` | Prepare (renders the transition buffer in the background) |
| `M` | Mix Now (starts the crossfade into the prepared buffer) |
| `B` | Backspin transition — abruptly stop the playing track, play the backspin SFX, then start the queued next track at natural tempo from its **raw** cue (`0:00` or the exact typed value, not the bar-snapped `Mix:` cue). Requires a track playing, a **raw** next track (queued but NOT prepared and NOT being prepared), and no in-progress crossfade/restoration. Rejected with a status message otherwise. |
| `Space` | Play / Pause |
| `S` | Stop |
| `Q` | Quit |
| `→` / `Enter` / click (on **root**) | Expand root to reveal subfolders; move highlight to first subfolder (focus stays on Tree) |
| `→` / `Enter` / click (on a **subfolder**) | Expand it, load its songs into the DataTable, and focus the DataTable |
| `↑` / `↓` (in Tree) | Move between folders; once subfolders are shown the highlight never returns to the root via arrows |
| `←` (in Tree) | Go up one level: collapse the parent folder and highlight it (subfolder display → root display in one press) |
| `←` (in DataTable) | Clear songs, collapse the browsed subfolder, return focus to the Tree |

## Architecture

The app is split into three layers that communicate in one direction: `main.py` → `analyzer` → `app` → `engine`.

**`automix/analyzer.py`** — Pure analysis, no audio playback. `analyze_library(root)` scans a folder tree recursively and returns `{abs_path: {bpm, key, beats, downbeats}}`. `librosa.beat.beat_track` provides BPM and beat frame indices; Krumhansl-Kessler chroma correlation provides the key label. Downbeats are detected heuristically: a single STFT pass produces a 40–150 Hz energy curve, and for each candidate phase offset `p ∈ {0,1,2,3}` the total energy at beats `p, p+4, p+8, …` is summed; the highest-scoring phase defines the downbeats (assumes 4/4). `beats` and `downbeats` are stored as integer sample indices at 44100 Hz (the engine's rate) so the app can use them directly against `engine.position` without unit conversion. Decoding for analysis is via pydub/ffmpeg at 22050 Hz mono. Cache lives at `~/.automix_cache.json` with schema `{"version": 2, "entries": {hash: record}}`; cache keys are an MD5 of `path:mtime:size`, and entries whose `version` doesn't match `CACHE_VERSION` are silently treated as missing and re-analyzed.

**`automix/audio_engine.py`** — Stateful playback engine built on a single persistent `sd.OutputStream` running at 44100 Hz stereo. The stream callback (`_callback`) runs on a real-time audio thread and must never do I/O — it reads from pre-loaded numpy arrays under a `threading.Lock`. Three states drive the callback: `IDLE` (silence), `PLAYING` (`_fill_playing`), `MIXING` (`_fill_mixing`). During `MIXING`, `_fill_mixing` applies a per-frame linear crossfade envelope between `_now_audio` and `_next_audio`, then transitions back to `PLAYING` when `_mix_pos >= _fade_samples`, promoting `_next_audio` to `_now_audio`. `start_mix()` accepts an optional `scheduled_start_sample`: when given, the engine stays in PLAYING with `_pending_mix_at` armed, and `_fill_playing` splits the chunk at that sample within a single callback — filling the pre-trigger frames from `_now_audio`, then flipping state to MIXING and delegating the remainder to `_fill_mixing` so there's no silence seam at the trigger point. The engine itself owns no time-stretching code — all stretching is done outside in `stretcher.py` before audio is handed to `play()` or `start_mix()`.

**`automix/stretcher.py`** — Thin wrapper around the `rubberband` CLI. `make_transition_buffer(audio, start_rate, fade_seconds, restore_seconds, sample_rate, progress_callback=None)` produces a single contiguous time-stretched buffer for the entire remainder of the incoming track: the first `fade_seconds` of output is at a constant `start_rate` (matches the outgoing track's BPM), the next `restore_seconds` smoothly returns to rate 1.0 along a **smootherstep** curve (`6x⁵ − 15x⁴ + 10x³`), and the rest is at rate 1.0 to the end of the track. Internally builds a 64-anchor timemap from the analytic integral of smootherstep (`S(x) = x⁶ − 3x⁵ + 2.5x⁴`), writes input/timemap to temp files, and shells out to `rubberband -3 --centre-focus -M <map> -D <duration_s> in.wav out.wav` (R3 fine engine; `-M` is the timemap flag, `-D` is the required overall duration; an explicit `(0,0)` timemap row triggers a rubberband NaN bug so the first anchor is always strictly past origin). Output is read back via `soundfile`. When `progress_callback` is supplied the call switches to `Popen`+a daemon thread that drains stderr character-by-character (rubberband uses CR-overwrite for the percent digits), parses `Pass 1:` / `Pass 2:` markers and `<n>%` lines, and forwards a 0.0–1.0 fraction. `rubberband_available()` is the startup probe used by `main.py`.

**`automix/transition.py`** — Pure (no rubberband, no engine state) skip-vs-stretch planner. `plan_transition(now_bpm, next_bpm, fade_seconds)` returns a `TransitionPlan(skip, start_rate, matched_bpm, drift_beats, relation)`. It octave-folds the raw ratio `r = now_bpm/next_bpm` by picking `r_eff = argmin over k ∈ {r/2, r, 2r} of |ln k|` (the interpretation closest to 1.0 in log space, capped at one octave), so half-/double-time pairs fold to ~1.0 instead of a literal 2× stretch. Projected end-of-fade beat slip if mixed unstretched is `drift_beats = (next_bpm/60)·|r_eff − 1|·fade_seconds` (for same-tempo this is the absolute-BPM-independent `|Δbpm|·fade/60`). `skip = drift_beats ≤ MAX_DRIFT_BEATS` (0.25, a quarter beat). `start_rate = r_eff` is what the STRETCH path feeds `make_transition_buffer` — this is the fix for the old `start_rate = now_bpm/next_bpm` bug that would speed a 64 BPM track to 128. Missing tempo on either side returns a skip plan at rate 1.0 (no division, no pointless render). The octave fold is factored into `fold_ratio(r) → (r_eff, relation)`, shared with `tempo_compatible(now_bpm, song_bpm, tolerance=TEMPO_MATCH_TOLERANCE)` — a fade-independent boolean (`|r_eff − 1| ≤ tolerance`, default 6%) that drives the song-list `:)` match gutter.

**`main.py`** — Entry point. Uses `argparse`: positional `music_folder` and optional `--backspin <path>` (default `samples/top_DJ_Rewind_SFX_10.mp3`; `_resolve_backspin_sample` resolves a bare name inside `samples/`). Hard-fails if the resolved backspin sample doesn't exist or if `rubberband` is missing from PATH, prints an ASCII progress bar while `analyze_library` runs, then launches `AutoMixApp(root, library, backspin_sample=...)`.

**`automix/app.py`** — Textual UI. `AutoMixApp` owns one `AudioEngine` instance. All file loading and rubberband rendering runs in `threading.Thread(daemon=True)` workers; results are pushed back to the UI via `self.call_from_thread(...)`. A 100 ms `set_interval` timer drives `NowPlayingPanel.refresh_progress`. The folder tree uses lazy expansion: subdirectories get a `"__placeholder__"` child node on first render; `on_tree_node_expanded` replaces it with real children. `_songs_in_view` is a list of absolute paths that stays in sync with the `DataTable` rows — `_selected_song()` uses `table.cursor_row` as an index into it. The DataTable's leading `Mix` column is a one-glyph gutter showing `:)` next to songs whose tempo is beat-mixable with the now-playing track (octave-folded via `tempo_compatible()`); `_refresh_match_markers()` recomputes it (via `DataTable.update_cell` keyed by path + the stored `_match_col_key`) whenever the now-playing track changes — `_do_load_now`, `_apply_now_swap`, `_handle_track_finished`, `action_stop` — and `_load_songs_for` seeds it when populating rows. `FolderTree` subclasses `Tree` and intercepts right/left/up/space in its own `on_key` (posting `FolderTree.GoToSongs` on a subfolder right-arrow) to drive the folder-navigation state machine — see the Navigation contract under Key invariants.

Tempo transition pipeline: `action_prepare_mix()` reads `cue`, `fade`, and `restore` from the NextTrackPanel and calls `plan_transition(now_bpm, next_bpm, fade)`. The returned `TransitionPlan` is stashed in `self._next_plan` (held in lockstep with `_next_prepared`) and branches the prepare into two paths. **STRETCH** (`not plan.skip`): in a background thread, call `make_transition_buffer(audio[cue:], plan.start_rate, fade, restore, ..., progress_callback=_on_progress)` — the entire remainder of Track B as a single continuous buffer with the rate-ramp (from `r_eff` back to 1.0) pre-baked in. **SKIP** (`plan.skip` — near-identical tempos, or an exact half-/double-time pair within the drift budget): `_prepare_skip()` runs a background load and stashes the **raw** `audio[cue:]` as `_next_prepared` with no rubberband call, no BPM animation. Either way the result lands in `self._next_prepared`; `_next_plan.skip` tells `action_mix_now` which kind it is. `action_mix_now()` then picks Track A's next downbeat after `engine.position + 100 ms` from `_now_downbeats`, and hands the buffer to `engine.start_mix(buf, fade, scheduled_start_sample=<that downbeat>)`. With the cue already snapped to a Track B downbeat by `NextTrackPanel.set_cue`, sample 0 of the prepared buffer = Track B's downbeat; firing it on Track A's downbeat makes the downbeats coincide → bar-aligned crossfade. If either track has no downbeat data, or the cue wasn't snapped, `scheduled_start_sample` is `None` and the engine starts the mix immediately (legacy behaviour, status line reflects this). There is no post-crossfade swap, no chunk boundaries, no second worker — by construction, the only seam in the output is the track-to-track crossfade itself. For a deferred (bar-aligned) mix, `action_mix_now` does NOT swap the now-playing metadata immediately — it stashes the incoming track's `(path, bpm, key, downbeats)` in `_pending_now_swap` and `_tick()` applies it via `_apply_now_swap()` only when it observes `PLAYING→MIXING` (the crossfade actually firing), so the NowPlaying panel keeps showing the outgoing track during the bar-wait. The immediate-mix path swaps synchronously. On a **SKIP** mix, `action_mix_now` leaves `_restore_from_bpm`/`_restore_to_bpm` at `0.0` (the incoming plays at its natural tempo throughout, so there is nothing to restore), which keeps `_tick` from arming any restoration ramp or overriding the now-playing BPM; on a **STRETCH** mix it sets `_restore_from_bpm = plan.matched_bpm` (= `next_bpm·r_eff`) and `_restore_to_bpm = next_bpm`. `_tick()` detects `MIXING→PLAYING` to arm the live BPM-display ramp (only when those endpoints differ, so a skip mix never arms it); `PLAYING→MIXING` while `_mix_scheduled` to apply the pending swap and flip the status line; and `PLAYING/MIXING→IDLE` (with `_now_path` still set) to run `_handle_track_finished()` — a natural track-end that clears now-playing, invalidates any prep (the buffer was matched to the finished track), keeps the queued NEXT track but marks it unprepared, and returns the app to a clean IDLE so `Enter` can load again.

Prepare-time BPM animation (STRETCH path only): the NextTrackPanel shows the displayed BPM sliding from `next_bpm` toward `plan.matched_bpm` (= `now_bpm` for a same-octave match) while Prepare runs. This is driven by `self._prep_progress` (0.0–1.0), updated by the stretcher's `progress_callback` from rubberband's own stderr — not by an elapsed-time clock. `_tick()` reads `_prep_progress`, applies smootherstep, and updates `NextTrackPanel._display_bpm`. By construction the animation lands at `matched_bpm` exactly when prep completes, regardless of track length or CPU speed. The SKIP path renders no rubberband, so there is no progress to animate — `_prepare_skip` just sets the panel status to a "no stretch" READY line.

Restoration BPM ramp & pause: the post-crossfade BPM ramp shown in `NowPlayingPanel` is wall-clock driven (`_t_restore_start`), but the underlying tempo restoration is baked into the buffer and only advances when audio plays. So `_tick()` freezes the ramp while `engine.paused` by pushing `_t_restore_start` forward by each tick's `dt` — keeping `(now - start)` constant across the pause so the displayed BPM doesn't run ahead of the frozen audio. `_last_tick_t` holds the previous tick's timestamp for the `dt` computation.

Changing C/F/R after Prepare calls `_mark_stale()` which clears `_next_prepared`, resets `_prep_progress`, and shows `[STALE - press P]` on the panel, forcing a re-Prepare before Mix can succeed. `NextTrackPanel._display_bpm` is a `_tick()`-driven override for the prep-animation BPM display; `_render()` uses it when non-`None`.

Backspin transition (`B`, `action_backspin`): the third transition mode alongside **SKIP** and **STRETCH**, but it bypasses `plan_transition()` entirely — no rubberband, no crossfade. It abruptly stops the outgoing track, plays a backspin/rewind SFX one-shot, then starts the queued next track at its natural tempo from its **raw** cue (`panel.raw_cue` — `0:00` or the exact typed value, NOT the bar-snapped cue the crossfade uses). Mechanically it builds a single buffer `np.concatenate([_backspin_audio, next_audio[cue_sample:]])` (where `cue_sample = int(raw_cue * SAMPLE_RATE)`) and hands it to `engine.play()` (played verbatim from index 0, like any other engine buffer). The SFX is decoded once at mount by `_preload_backspin()` into `self._backspin_audio` (a daemon thread, so the keypress has no decode hitch). By design it requires a **raw** next track: `action_backspin` rejects the press unless a track is playing, a next track is queued, `not self._preparing`, `self._next_prepared is None`, and the engine is not mid-transition (`state != MIXING` and `_t_restore_start == 0.0`) — a prepared/stretched buffer is irrelevant to the SFX-then-play mechanic. The worker (`load_audio` + concatenate) hands off to `_apply_backspin()` on the UI thread, which plays the buffer, promotes the next track to now-playing, arms NO restoration ramp (natural tempo, like a SKIP), clears the NEXT slot, and offsets `_now_downbeats` by `len(_backspin_audio) - cue_sample` (dropping pre-cue downbeats) so a subsequent bar-aligned mix still lines up. Because the engine was already PLAYING and stays PLAYING, `_tick`'s MIXING→PLAYING / →IDLE detectors don't fire — no spurious ramp or `_handle_track_finished`.

## Key invariants

- Navigation contract: the folder browser is a three-level drill-down — **root display** (root node collapsed, song panel empty; this is the startup state, set by `_build_tree` which leaves the root collapsed and `on_mount` which no longer pre-loads songs) → **subfolder display** (root expanded, subfolders visible) → **song browsing** (a subfolder expanded + its songs loaded in the DataTable + focus on the DataTable). The three activation gestures — right arrow, Enter, mouse click on a node's label — drive each step: on the **root** node they only expand it (reveal subfolders, move the highlight onto the first subfolder, focus stays on the Tree); on a **subfolder** node they expand it AND load its songs AND move focus to the DataTable (right arrow via the `FolderTree.GoToSongs` message; Enter/click via `Tree.NodeSelected` → `on_tree_node_selected`). Left arrow walks back up one level: in the DataTable it clears the song list, collapses the browsed subfolder, and returns focus to the Tree; in the Tree (`FolderTree.on_key`) it collapses the cursor node's **parent** and moves the highlight onto it — so a single press from a first-level subfolder collapses the root and lands the highlight back on it (subfolder display → root display) in one step. The root is never re-highlighted by up/down navigation: `on_key` intercepts `up` and blocks the step that would land on the root (`get_node_at_line(cursor_line-1) is root`), so once subfolders are shown the arrows only walk among them. `FolderTree` sets `auto_expand = False` so Enter/click never auto-toggle a node — expansion is driven only by the handlers above; `on_key` also intercepts `space` (prevent_default to block Tree's built-in toggle) and forwards it to the global `action_toggle_pause` so spacebar never flips a folder's arrow while browsing. Because root activation never loads songs, songs living directly in the root folder (not in a subfolder) are not browsable by design. Do not add app-level left/right bindings — they would shadow widget-level handling.
- All mutations to `AudioEngine` state (`_now_audio`, `_next_audio`, `_position`, `_mix_pos`, `state`) must be done inside `self._lock`. The callback also acquires the lock, so keep critical sections short.
- `_next_audio` passed to `start_mix()` must already be rendered by `make_transition_buffer()` (or be the full unmodified track for a non-tempo-matched cue) — the engine plays it verbatim from index 0.
- After `MIXING → PLAYING` transition, `_position` is set to `_mix_pos` (not 0), so playback of the former next track continues from where the crossfade left off.
- `call_from_thread` is the only safe way to touch Textual widgets from the audio/prep worker threads.
- The inline cue/fade text input (`_input_mode`) intercepts keys in `on_key` and calls `event.prevent_default()` + `event.stop()` to suppress bindings while active.
- `make_transition_buffer()` is the **only** place in the project that invokes rubberband. All time-stretching routes through it. The librosa import in `audio_engine.py` is gone; librosa is now used only by `analyzer.py` for BPM/key detection.
- Any change to Cue / Fade / Restore after Prepare must call `_mark_stale()`. The precomputed buffer's contents depend on all three; silently keeping a stale buffer was a pre-rubberband bug and must not return.
- The stretch rate is **always** `plan.start_rate` (the octave-folded `r_eff`), never the raw `now_bpm / next_bpm`. Computing it raw reintroduces the half-/double-time bug (a 64 BPM track stretched to 128). All callers go through `plan_transition()`; do not recompute the ratio by hand. `_next_plan` is held in lockstep with `_next_prepared` — reset BOTH to `None` together everywhere (`action_stop`, `_handle_track_finished`, `action_load_next`, `_mark_stale`, and after consumption in `action_mix_now`). A `_next_plan.skip == True` buffer is the **raw** cue audio (no rubberband); `action_mix_now` must arm no restoration ramp for it (`_restore_from_bpm`/`_restore_to_bpm` left at `0.0`), because a skipped track plays at its natural tempo with nothing to restore. `MAX_DRIFT_BEATS` (0.25) in `transition.py` is the single tunable that defines "tempos close enough to skip"; the implied BPM window scales inversely with fade length (`|Δbpm| ≤ 60·MAX_DRIFT_BEATS/fade`).
- `action_mix_now()` is blocked while `engine.state == MIXING` or `_t_restore_start > 0.0` (i.e. previous transition's crossfade or restoration ramp still running). The prepared buffer's `start_rate` is rendered against `_now_bpm` (the natural BPM of what's playing), which only equals the *actual* playback BPM once any prior ramp has finished. Preparing during a transition is fine; mixing into one is not. The block messages are phase-specific ("Mixing the two tracks — cannot mix another track yet." vs "Restoring original tempo — cannot mix another track yet.") and match the `NowPlayingPanel._phase` banner that `_tick()` keeps in sync with the same two conditions. `_tick()` sets the banner to one of four states whenever a track is playing (`state != IDLE`): "MIXING the two tracks - cannot mix another track" during the crossfade, "WAITING for downbeat - cannot mix another track" while `_mix_scheduled` (a bar-aligned mix is armed but the trigger downbeat hasn't arrived yet), "RESTORING original tempo - cannot mix another track" while `_t_restore_start > 0.0`, and otherwise "Ready to mix another song" — i.e. the idle-of-transition state where `action_mix_now` would be accepted. During the crossfade only, the panel title shows both tracks as "outgoing → incoming": `_apply_now_swap` passes the outgoing name to `NowPlayingPanel.set_track(mix_from=...)` (captured before `_now_path` is overwritten), and `_tick` calls `panel.clear_mix_from()` at MIXING→PLAYING so the dual name reverts to the single incoming track once the crossfade ends (`clear()`/`set_track` without `mix_from` also reset it, covering Stop / track-end / load paths). The progress line mirrors this: during MIXING, `_tick` passes the engine's `mix_position`/`mix_duration` (the incoming buffer's playhead/length — `_mix_pos` and `len(_next_audio)`, exposed as engine properties) to `refresh_progress`, which renders two 18-wide bars "outgoing → incoming" side by side via `_progress_segment()`; outside MIXING it passes `None`/`None` and falls back to the single 32-wide bar. Note the incoming buffer is the whole remaining track, so its bar sits near-empty during the short fade window (accurate, not a bug). Both bars MUST build their `[...]` through `_progress_segment`, whose leading bracket is escaped `\[` — an unescaped `[#...` is parsed by Rich as a colour tag and the bar silently vanishes (see the Rich-markup gotcha).
- Now-playing track changes ONLY through `Enter` (when engine IDLE) or through a completed Mix; pressing `Enter` while anything is playing is rejected by `on_data_table_row_selected`. This means `_do_load_now` is never called against a non-IDLE engine via the UI — keep that invariant when wiring any new callers.
- `action_prepare_mix` and `action_mix_now` both require the engine to be actually playing (`state != IDLE`, and for Prepare `_now_bpm > 0`). Prepare's plan rate `r_eff` (from `now_bpm / next_bpm`) is degenerate with nothing playing; Mix's `engine.start_mix` no-ops on a non-PLAYING engine, so without the guard the app would desync (clear the NEXT panel and claim "Mixing…" while the engine produced silence). Keep both guards if you add new entry points.
- When the engine reaches IDLE on its own (track played to the end, not an explicit Stop), `_tick()` must call `_handle_track_finished()` exactly once — guarded by `_now_path is not None` so it doesn't re-fire after `action_stop` (which already nulls `_now_path`). The handler invalidates prep via `_prep_epoch` because any prepared buffer's `start_rate` was rendered against the track that just ended.
- `_do_load_now()` (Enter on a song row) must clear `_t_restore_start`, `_restore_from_bpm`, `_restore_to_bpm`, `_mix_scheduled`, AND bump `_prep_epoch` (plus clear `_preparing`/`_prep_animating`/`_prep_progress`) before spawning its load worker. Otherwise a prior transition's BPM-display ramp continues animating against the freshly loaded track, and — if Enter interrupted an active crossfade — the next `_tick()` would re-arm the ramp via the MIXING→PLAYING detector. Also, an in-flight Prepare worker was rendering against the OLD `_now_bpm`; without the epoch bump it would install a wrong-tempo buffer when it completes. The engine's `play()` only swaps `_now_audio`; it doesn't know anything about app-level restoration or prep state.
- `_prep_epoch` is the single source of truth for "is the current prep worker still relevant?". Bump it (and ideally clear `_preparing` / `_prep_animating` / `_prep_progress`) in EVERY action that invalidates an in-flight prep: `action_load_next` (queued next-track changed), `action_prepare_mix` (new prep supersedes any old one), `_mark_stale` (C/F/R changed → old buffer's parameters wrong), `action_stop`, `_do_load_now` (now-playing changed → `r_eff` denominator wrong), and `_handle_track_finished` (track ended → buffer matched a track no longer playing). Each worker captures the epoch at launch and aborts on completion if it differs. NOTE the epoch only orphans *in-flight* workers — actions that can leave a **completed** `_next_prepared` buffer dangling against a no-longer-current now-playing track (`action_stop`, `_handle_track_finished`) must ALSO set `_next_prepared = None` (and `_next_plan = None`) and reset the NEXT panel to `[not prepared]`, or a later `Enter`→`M` would mix a wrong-tempo buffer.
- `NextTrackPanel` keeps **two** cue values in seconds, both written only by `set_cue()`: `_cue_raw` is the unsnapped value (`0.0` or exactly what the user typed; exposed as `raw_cue`), and `_cue` is the bar-snapped version (nearest downbeat when `_downbeats` is non-empty, else identical to raw; exposed as `cue`, flag `cue_snapped`). **Backspin (`action_backspin`) reads `raw_cue`; Prepare/Mix (`action_prepare_mix`, `action_mix_now`) read the snapped `cue`** — so a `0:00` cue backspins from the track start while the crossfade still bar-aligns. Any new code that writes `_cue`/`_cue_raw` directly (bypassing `set_cue`) breaks this split and the bar alignment. The panel renders both as `Cue: <raw>   Mix: <snapped> [bar]` (the `Mix:` field only when snapped). The `cue_snapped` flag is the source of truth `action_mix_now` uses to decide whether bar-aligned scheduling is applicable.
- `action_mix_now` only schedules a deferred (bar-aligned) crossfade when BOTH tracks have downbeats AND the cue was snapped. With a half-aligned pair (only one side has downbeats), scheduling would delay the mix without making it any more musical, so it falls back to immediate start. Status line reports which path was taken.
- The backspin transition (`action_backspin`, `B`) requires a **raw** next track and bypasses the prepare pipeline. Its guards — track playing, `_next_path` set, `not _preparing`, `_next_prepared is None`, `state != MIXING`, `_t_restore_start == 0.0`, `_backspin_audio` loaded — must all hold; a prepared OR being-prepared next track is explicitly rejected (the SFX-then-play mechanic never uses the prepared buffer). It starts the next track from the **raw** cue (`panel.raw_cue`, NOT the snapped `cue`), arms no restoration ramp (leave `_restore_from_bpm`/`_restore_to_bpm` at `0.0`, like a SKIP), and MUST offset `_now_downbeats` by `len(_backspin_audio) - cue_sample` (prepended SFX + raw-cue cut), dropping pre-cue downbeats, or a later bar-aligned mix lands off the bar. It calls `engine.play()` (not `start_mix`), so the engine stays PLAYING and `_tick` arms nothing — do not route it through the MIXING transition detectors.
- The library dict that the app and analyzer share has the shape `{path: {bpm, key, beats, downbeats}}` (a per-track record dict, NOT a `(bpm, key)` tuple). Use `empty_record()` from `analyzer.py` as the default in `library.get(path, ...)` calls; missing/garbled cache entries become an empty record rather than a KeyError.

## Gotchas

**Textual version ceiling** — `requirements.txt` pins `textual<1.0.0`. Textual 1.x+ (now at 8.x) rewrote internal rendering APIs (`Visual`, `Content`, `render_strips`) in ways that break `Static` subclasses. Do not bump past `0.x`.

**Rich markup in `Static.update()`** — Textual passes strings through Rich markup parsing. Any literal `[`, `]` in displayed text (e.g. `[none]`, `[C]`, filenames) must be escaped as `\\[`. Use `rich.markup.escape()` for user-supplied strings (filenames). Failure produces `AttributeError: 'NoneType' object has no attribute 'render_strips'` at render time.

**WSL audio** — `sounddevice` has no audio devices under WSL without WSLg (WSL2 + `wsl --update`) or manual PulseAudio-over-TCP setup. Run natively on Windows or a Linux desktop instead.

**Progress bar is ASCII** — `main.py` uses `#` and `-` (not `█░`) because Windows terminals default to cp1252, which cannot encode Unicode block characters and raises `UnicodeEncodeError` before the TUI starts.

**Textual key event order** — Within a widget, `on_key` fires *before* the widget's own BINDINGS dispatch; `event.prevent_default()` + `event.stop()` in a subclass `on_key` successfully blocks built-in key actions. App-level `on_key` fires *after* the focused widget has already acted, so intercepting a key before Tree or DataTable handles it requires subclassing the widget, not handling it in the App. Also: `Tree.NodeSelected` fires only on Enter/click, not on arrow-key navigation; use `Tree.NodeHighlighted` for cursor-movement events.

**Rubberband stderr parsing** — Rubberband uses **carriage-return overwrite** for its `<n>%` progress digits (not newlines), so `readline()` and the default line-buffered iterator both block until the pass ends, defeating real-time progress. `_drain_stderr_with_progress` reads stderr one character at a time and chunks on either `\r` or `\n`. Also: in R3 mode Pass 1 ("Studying") is essentially instant (~1% of wall time) and Pass 2 ("Processing") does ~99% — a naive 50/50 weighting causes the BPM display to snap to halfway in the first frame and crawl the remainder over many seconds. The constants `_PASS1_WEIGHT = 0.05` / `_PASS2_WEIGHT = 0.95` in `stretcher.py` reflect this asymmetry; if you ever switch to the R2 engine these need re-calibrating.

**Rubberband CLI flag trap** — `-T` is `--tempo` (a constant stretch factor), NOT the timemap flag. The timemap flag is `-M` (or `--timemap`), and when used it requires an overall stretch factor alongside it, supplied via `-D <duration_seconds>` (or `-t`/`-T`). Without `-D`, rubberband silently produces input-length output with no stretching applied. Also: an explicit `(0, 0)` first row in the timemap triggers a NaN-time-ratio computation at sample 0 that corrupts output; rubberband anchors `(0, 0)` implicitly, so the first explicit anchor must be strictly past origin (handled by `_build_timemap`'s monotonic filter).

**Downbeat detection is heuristic** — `analyzer.py` runs librosa's `beat_track` then assigns downbeats by picking the phase (`0,1,2,3`) that maximises summed 40–150 Hz STFT energy at every 4th beat. Works well on 4/4 dance/pop (consistent kick on the 1); less reliable on jazz, 3/4 material, or tracks with no clear kick drum. A track with no detected beats (atonal/very short) gets empty `beats`/`downbeats` lists, and any mix involving it falls back to immediate (non-aligned) start — `action_mix_now` makes this decision explicitly.

**madmom on Python 3.12 is impractical** — Tried madmom (more accurate downbeats) and BeatNet (which also needs madmom). Madmom's PyPI release is from 2018, uses `np.float` (removed in numpy 1.24), and requires MSVC C++ build tools to compile. Going that route demands pinning `numpy<2` + installing Build Tools. The librosa heuristic is the conscious trade-off for keeping the deps simple.

**Cache schema versioning** — The cache file at `~/.automix_cache.json` is now `{"version": 2, "entries": {...}}`. The loader rejects any other shape and re-analyzes from scratch — so upgrading the schema is automatic but the first launch after a bump pays the full re-analysis cost. Bump `CACHE_VERSION` whenever the per-track record's required fields change.

## Audio data format

All audio arrays are `(N, 2) float32` in `[-1.0, 1.0]`, at 44100 Hz. pydub decodes to 16-bit PCM integers (`/ 32768.0` normalises). Analysis uses a separate 22050 Hz mono path inside `analyzer.py` and never shares arrays with the engine.

## Testing

No pytest/unittest framework — `tests/` holds standalone probe scripts, each run directly and printing `PASS`/`All ... passed`. Run the whole suite:

```powershell
Get-ChildItem tests\probe_*.py | ForEach-Object { .\.venv\Scripts\python.exe $_.FullName }
```

| Script | Covers |
|---|---|
| `probe_analyzer.py` | Downbeat detection on real tracks (needs files under `music/`) |
| `probe_engine_split.py` | `start_mix` scheduled-start chunk-split (sample-accurate) |
| `probe_prep_epoch.py` | Prep-supersession races (N / P / stale / Enter during Prepare) |
| `probe_cue_reset.py` | Cue snap + reset-on-N behaviour |
| `probe_phase_banner.py` | `NowPlayingPanel` phase-line rendering + "outgoing → incoming" dual name during a crossfade |
| `probe_conflicts.py` | Track-end cleanup, P/M guards, deferred swap, pause-during-ramp, Stop, self-mix |
| `probe_transition_plan.py` | `plan_transition` skip-vs-stretch decision, octave folding, drift formula, half-time bug guard |
| `probe_skip_mix.py` | Skip path stores raw buffer + arms no restore ramp; stretch path arms ramp from `matched_bpm` |
| `probe_backspin.py` | Backspin `B`: guard rejections (idle/no-next/preparing/prepared/mixing/restoring/unloaded), `[SFX + cue]` buffer build, now-playing swap, no ramp, downbeat offset |
| `probe_e2e_alignment.py` | Full bar-alignment against real tracks (needs `music/`) |

**Headless harness pattern** — the engine and Textual UI are tested without an audio device or a mounted app: build the `AudioEngine` via `AudioEngine.__new__(...)` and set its state fields manually (skips opening the `sd.OutputStream`), then drive `_callback`/`_fill_playing` directly with synthetic numpy audio. For app-level logic, build `AutoMixApp.__new__(...)`, stub `query_one` to return fake panels and `_status` to record messages, and call `_tick()` / action methods directly. Keep this pattern when adding tests — a real stream or mount can't run in CI/headless.

`tests/diagnose_progress.py` is not a test — it's a diagnostic that runs `make_transition_buffer` on a real track and logs every rubberband stderr chunk + parsed progress to `tests/progress_log.txt` (used to debug the prep-progress oscillation bug).
