# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Quick start

**Linux / macOS**
```bash
sudo apt install ffmpeg libportaudio2 rubberband-cli   # Debian/Ubuntu; use dnf/pacman equivalents elsewhere
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python main.py /path/to/music/folder
```

**Windows** (run in Windows Terminal, not cmd.exe)
```powershell
winget install Gyan.FFmpeg              # sounddevice bundles PortAudio ‚Äî no separate install needed
choco install rubberband                # OR download from https://breakfastquay.com/rubberband/ and add to PATH
python -m venv .venv; .venv\Scripts\activate
pip install -r requirements.txt
python main.py C:\path\to\music
```

`main.py` checks for the `rubberband` CLI on PATH at startup and refuses to launch without it ‚Äî every tempo transition is rendered through it.

The backspin transition (`B`) uses a SFX one-shot, default `samples/top_DJ_Rewind_SFX_10.mp3`, overridable via `--backspin <path>`: a bare filename is resolved inside `samples/`, a relative/absolute path is used as-is (`python main.py <music_folder> --backspin backspin_09.wav`). The resolved file must exist or the app hard-fails at startup.

Output routing ‚Äî two independent streams, each pinnable to a device by name substring (matched case-insensitively against **output-capable** devices only, so a headset's mono-mic profile sharing the name is skipped; first match wins; an unknown name hard-fails listing the available outputs). Run `python main.py --list-devices` to print the output-capable devices and their indices (do it **with any AUX cable already plugged in**, since the jack can change the built-in codec's device naming).

- `--main-device "<name>"` pins the **master** mix to a chosen output instead of the flaky OS default ‚Äî the speakers, or a **mixer/PA via the built-in codec's AUX (3.5 mm) jack** (plug the cable into the jack and point `--main-device` at the Realtek/built-in codec; the codec auto-routes to the jack). Omit to follow the OS default.
- `--headphones-device "<name>"` sends the pre-listen / PFL cue (`L`, audition the queued NEXT track) to headphones. **Must be a SEPARATE device ‚Äî USB-C or Bluetooth** ‚Äî NOT the built-in 3.5 mm jack, which shares the master/speaker codec. Omit to disable cueing. (CLI flag only; the feature is "cue/PFL" internally ‚Äî `CuePlayer`, `action_cue_toggle`.)

The classic DJ setup is `python main.py <music> --main-device "Realtek" --headphones-device "B01"` ‚Üí master out to speakers/mixer, cue in the headphones. The master and headphones must resolve to **different** outputs or the app hard-fails ‚Äî including the case where `--main-device` is **omitted** and the OS default output happens to equal the headphones device (the master falls back to the OS default, so it would land on the headphones too). The guard compares the headphones device against the *effective* master (`--main-device`, or `sd.default.device[1]` when omitted).

On first run the app blocks at the terminal while analyzing every audio file; subsequent runs are instant because results are cached at `~/.automix_cache.json`.

## Architecture

The app is split into three layers that communicate in one direction: `main.py` ‚Üí `analyzer` ‚Üí `app` ‚Üí `engine`.

**`automix/analyzer.py`** ‚Äî Pure analysis, no audio playback. `analyze_library(root)` scans a folder tree recursively and returns `{abs_path: {bpm, key, beats, downbeats}}`. `librosa.beat.beat_track` provides BPM and beat frame indices; Krumhansl-Kessler chroma correlation provides the key label. Downbeats are detected heuristically: a single STFT pass produces a 40‚Äì150 Hz energy curve, and for each candidate phase offset `p ‚àà {0,1,2,3}` the total energy at beats `p, p+4, p+8, ‚Ä¶` is summed; the highest-scoring phase defines the downbeats (assumes 4/4). `beats` and `downbeats` are stored as integer sample indices at 44100 Hz (the engine's rate) so the app can use them directly against `engine.position` without unit conversion. Decoding for analysis is via pydub/ffmpeg at 22050 Hz mono. Cache lives at `~/.automix_cache.json` with schema `{"version": 2, "entries": {hash: record}}`; cache keys are an MD5 of `path:mtime:size`, and entries whose `version` doesn't match `CACHE_VERSION` are silently treated as missing and re-analyzed.

**`automix/audio_engine.py`** ‚Äî Stateful playback engine built on a single persistent `sd.OutputStream` running at 44100 Hz stereo (opened on `AudioEngine(device=...)`; `device=None` follows the OS default, an explicit index pins the master output ‚Äî set by `main.py`'s `--main-device`). A software master gain `_volume` (`set_volume`/`volume`, the `,`/`.` keys) is applied to `outdata` at the end of `_callback`, so the keyboard controls the level even when the master is pinned to a non-default device (Windows volume keys only act on the OS default device). The gain clamps to `[0.0, MAX_GAIN]` (`MAX_GAIN = 2.0`, +6 dB) ‚Äî above 1.0 it boosts quiet tracks, and the callback hard-limits the result to ¬±1.0 so a boosted signal never exceeds full scale at the device (speaker safety). Hardware media keys (Volume Up/Down) are intentionally not captured ‚Äî they're consumer-control HID keys the OS routes to the default device and never reach the TUI; capturing them would need a disruptive global keyboard hook. The zero-code alternative is to set the Windows default output to the master device. The stream callback (`_callback`) runs on a real-time audio thread and must never do I/O ‚Äî it reads from pre-loaded numpy arrays under a `threading.Lock`. Three states drive the callback: `IDLE` (silence), `PLAYING` (`_fill_playing`), `MIXING` (`_fill_mixing`). During `MIXING`, `_fill_mixing` applies a per-frame linear crossfade envelope between `_now_audio` and `_next_audio`, then transitions back to `PLAYING` when `_mix_pos >= _fade_samples`, promoting `_next_audio` to `_now_audio`. `start_mix()` accepts an optional `scheduled_start_sample`: when given, the engine stays in PLAYING with `_pending_mix_at` armed, and `_fill_playing` splits the chunk at that sample within a single callback ‚Äî filling the pre-trigger frames from `_now_audio`, then flipping state to MIXING and delegating the remainder to `_fill_mixing` so there's no silence seam at the trigger point. The engine itself owns no time-stretching code ‚Äî all stretching is done outside in `stretcher.py` before audio is handed to `play()` or `start_mix()`.

**`automix/stretcher.py`** ‚Äî Thin wrapper around the `rubberband` CLI. `make_transition_buffer(audio, start_rate, fade_seconds, restore_seconds, sample_rate, progress_callback=None)` produces a single contiguous time-stretched buffer for the entire remainder of the incoming track: the first `fade_seconds` of output is at a constant `start_rate` (matches the outgoing track's BPM), the next `restore_seconds` smoothly returns to rate 1.0 along a **smootherstep** curve (`6x‚Åµ ‚àí 15x‚Å¥ + 10x¬≥`), and the rest is at rate 1.0 to the end of the track. Internally builds a 64-anchor timemap from the analytic integral of smootherstep (`S(x) = x‚Å∂ ‚àí 3x‚Åµ + 2.5x‚Å¥`), writes input/timemap to temp files, and shells out to `rubberband -3 --centre-focus -M <map> -D <duration_s> in.wav out.wav` (R3 fine engine; `-M` is the timemap flag, `-D` is the required overall duration; an explicit `(0,0)` timemap row triggers a rubberband NaN bug so the first anchor is always strictly past origin). Output is read back via `soundfile`. When `progress_callback` is supplied the call switches to `Popen`+a daemon thread that drains stderr character-by-character (rubberband uses CR-overwrite for the percent digits), parses `Pass 1:` / `Pass 2:` markers and `<n>%` lines, and forwards a 0.0‚Äì1.0 fraction. `rubberband_available()` is the startup probe used by `main.py`.

**`automix/transition.py`** ‚Äî Pure (no rubberband, no engine state) skip-vs-stretch planner. `plan_transition(now_bpm, next_bpm, fade_seconds)` returns a `TransitionPlan(skip, start_rate, matched_bpm, drift_beats, relation)`. It octave-folds the raw ratio `r = now_bpm/next_bpm` by picking `r_eff = argmin over k ‚àà {r/2, r, 2r} of |ln k|` (the interpretation closest to 1.0 in log space, capped at one octave), so half-/double-time pairs fold to ~1.0 instead of a literal 2√ó stretch. Projected end-of-fade beat slip if mixed unstretched is `drift_beats = (next_bpm/60)¬∑|r_eff ‚àí 1|¬∑fade_seconds` (for same-tempo this is the absolute-BPM-independent `|Œîbpm|¬∑fade/60`). `skip = drift_beats ‚â§ MAX_DRIFT_BEATS` (0.25, a quarter beat). `start_rate = r_eff` is what the STRETCH path feeds `make_transition_buffer` ‚Äî this is the fix for the old `start_rate = now_bpm/next_bpm` bug that would speed a 64 BPM track to 128. Missing tempo on either side returns a skip plan at rate 1.0 (no division, no pointless render). The octave fold is factored into `fold_ratio(r) ‚Üí (r_eff, relation)`, shared with `tempo_compatible(now_bpm, song_bpm, tolerance=TEMPO_MATCH_TOLERANCE)` ‚Äî a fade-independent boolean (`|r_eff ‚àí 1| ‚â§ tolerance`, default 6%) that drives the song-list `:)` match gutter.

**`automix/cue_player.py`** ‚Äî `CuePlayer`, the pre-listen / PFL monitor: an independent second `sd.OutputStream(device=...)` on a separate device (USB-C / Bluetooth headphones) for auditioning the queued NEXT track while the master `AudioEngine` keeps playing on the speakers. Deliberately **decoupled from `AudioEngine`** ‚Äî its own stream, lock, buffer and playhead, no MIXING state machine, and it never shares state with the engine. `play(audio)` accepts an `(N,2)` float32 buffer at the engine's 44100 Hz (the app slices it at the **raw** cue, so index 0 = the drop point), `seek(¬±s)` moves the playhead clamped to `[0, end]`, and the callback stops **silent at end (no loop)**. At construction it opens at 44100; on `sd.PortAudioError` it retries at the device's `default_samplerate` and resamples each buffer to that rate via a cheap per-channel `np.interp` (`_resample`, no scipy). A software gain `_volume` (`set_volume`/`volume`, the `9`/`0` keys) is applied in the callback. **All PortAudio failures are swallowed internally** ‚Äî a device that vanishes mid-set (BT drop / USB unplug) sets `is_dead`, the player goes silent, and the failure never propagates into the master engine or the UI thread.

**`main.py`** ‚Äî Entry point. Uses `argparse`: optional positional `music_folder` (`nargs="?"` so `--list-devices` works without it), optional `--backspin <path>` (default `samples/top_DJ_Rewind_SFX_10.mp3`; `_resolve_backspin_sample` resolves a bare name inside `samples/`), and `--main-device` / `--headphones-device <name>` (both resolved by the shared `_resolve_output_device`, which substring-matches output-capable devices ‚Üí index, raising on no match; `_output_device_lines` backs both its error message and `--list-devices`). `--list-devices` prints the output devices and exits. Hard-fails if the backspin sample doesn't exist, if either device flag matched nothing, if the headphones device equals the **effective** master device ‚Äî the explicit `--main-device`, or the OS default output (`sd.default.device[1]`) when `--main-device` is omitted (both on one output defeats the split), or if `rubberband` is missing from PATH; prints an ASCII progress bar while `analyze_library` runs, then launches `AutoMixApp(root, library, backspin_sample=..., cue_device=..., main_device=...)` (each device `None` when its flag is absent ‚Üí master follows the OS default, cueing dormant).

**`automix/app.py`** ‚Äî Textual UI. `AutoMixApp` owns one `AudioEngine` instance. All file loading and rubberband rendering runs in `threading.Thread(daemon=True)` workers; results are pushed back to the UI via `self.call_from_thread(...)`. A 100 ms `set_interval` timer drives `NowPlayingPanel.refresh_progress`. The folder tree uses lazy expansion: subdirectories get a `"__placeholder__"` child node on first render; `on_tree_node_expanded` replaces it with real children. `_songs_in_view` is a list of absolute paths that stays in sync with the `DataTable` rows ‚Äî `_selected_song()` uses `table.cursor_row` as an index into it. The DataTable's leading `Mix` column is a one-glyph gutter showing `:)` next to songs whose tempo is beat-mixable with the now-playing track (octave-folded via `tempo_compatible()`); `_refresh_match_markers()` recomputes it (via `DataTable.update_cell` keyed by path + the stored `_match_col_key`) whenever the now-playing track changes ‚Äî `_do_load_now`, `_apply_now_swap`, `_handle_track_finished`, `action_stop` ‚Äî and `_load_songs_for` seeds it when populating rows. `FolderTree` subclasses `Tree` and intercepts right/left/up/space in its own `on_key` (posting `FolderTree.GoToSongs` on a subfolder right-arrow) to drive the folder-navigation state machine ‚Äî see the Navigation contract under Key invariants.

Tempo transition pipeline: `action_prepare_mix()` reads `cue`, `fade`, and `restore` from the NextTrackPanel and calls `plan_transition(now_bpm, next_bpm, fade)`. The returned `TransitionPlan` is stashed in `self._next_plan` (held in lockstep with `_next_prepared`) and branches the prepare into two paths. **STRETCH** (`not plan.skip`): in a background thread, call `make_transition_buffer(audio[cue:], plan.start_rate, fade, restore, ..., progress_callback=_on_progress)` ‚Äî the entire remainder of Track B as a single continuous buffer with the rate-ramp (from `r_eff` back to 1.0) pre-baked in. **SKIP** (`plan.skip` ‚Äî near-identical tempos, or an exact half-/double-time pair within the drift budget): `_prepare_skip()` runs a background load and stashes the **raw** `audio[cue:]` as `_next_prepared` with no rubberband call, no BPM animation. Either way the result lands in `self._next_prepared`; `_next_plan.skip` tells `action_mix_now` which kind it is. `action_mix_now()` then picks Track A's next downbeat after `engine.position + 100 ms` from `_now_downbeats`, and hands the buffer to `engine.start_mix(buf, fade, scheduled_start_sample=<that downbeat>)`. With the cue already snapped to a Track B downbeat by `NextTrackPanel.set_cue`, sample 0 of the prepared buffer = Track B's downbeat; firing it on Track A's downbeat makes the downbeats coincide ‚Üí bar-aligned crossfade. If either track has no downbeat data, or the cue wasn't snapped, `scheduled_start_sample` is `None` and the engine starts the mix immediately (legacy behaviour, status line reflects this). There is no post-crossfade swap, no chunk boundaries, no second worker ‚Äî by construction, the only seam in the output is the track-to-track crossfade itself. For a deferred (bar-aligned) mix, `action_mix_now` does NOT swap the now-playing metadata immediately ‚Äî it stashes the incoming track's `(path, bpm, key, downbeats)` in `_pending_now_swap` and `_tick()` applies it via `_apply_now_swap()` only when it observes `PLAYING‚ÜíMIXING` (the crossfade actually firing), so the NowPlaying panel keeps showing the outgoing track during the bar-wait. The immediate-mix path swaps synchronously. On a **SKIP** mix, `action_mix_now` leaves `_restore_from_bpm`/`_restore_to_bpm` at `0.0` (the incoming plays at its natural tempo throughout, so there is nothing to restore), which keeps `_tick` from arming any restoration ramp or overriding the now-playing BPM; on a **STRETCH** mix it sets `_restore_from_bpm = plan.matched_bpm` (= `next_bpm¬∑r_eff`) and `_restore_to_bpm = next_bpm`. `_tick()` detects `MIXING‚ÜíPLAYING` to arm the live BPM-display ramp (only when those endpoints differ, so a skip mix never arms it); `PLAYING‚ÜíMIXING` while `_mix_scheduled` to apply the pending swap and flip the status line; and `PLAYING/MIXING‚ÜíIDLE` (with `_now_path` still set) to run `_handle_track_finished()` ‚Äî a natural track-end that clears now-playing, invalidates any prep (the buffer was matched to the finished track), keeps the queued NEXT track but marks it unprepared, and returns the app to a clean IDLE so `Enter` can load again.

Prepare-time BPM animation (STRETCH path only): the NextTrackPanel shows the displayed BPM sliding from `next_bpm` toward `plan.matched_bpm` (= `now_bpm` for a same-octave match) while Prepare runs. This is driven by `self._prep_progress` (0.0‚Äì1.0), updated by the stretcher's `progress_callback` from rubberband's own stderr ‚Äî not by an elapsed-time clock. `_tick()` reads `_prep_progress`, applies smootherstep, and updates `NextTrackPanel._display_bpm`. By construction the animation lands at `matched_bpm` exactly when prep completes, regardless of track length or CPU speed. The SKIP path renders no rubberband, so there is no progress to animate ‚Äî `_prepare_skip` just sets the panel status to a "no stretch" READY line.

Restoration BPM ramp & pause: the post-crossfade BPM ramp shown in `NowPlayingPanel` is wall-clock driven (`_t_restore_start`), but the underlying tempo restoration is baked into the buffer and only advances when audio plays. So `_tick()` freezes the ramp while `engine.paused` by pushing `_t_restore_start` forward by each tick's `dt` ‚Äî keeping `(now - start)` constant across the pause so the displayed BPM doesn't run ahead of the frozen audio. `_last_tick_t` holds the previous tick's timestamp for the `dt` computation.

Changing C/F/R after Prepare calls `_mark_stale()` which clears `_next_prepared`, resets `_prep_progress`, and shows `[STALE - press P]` on the panel, forcing a re-Prepare before Mix can succeed. `NextTrackPanel._display_bpm` is a `_tick()`-driven override for the prep-animation BPM display; `_render()` uses it when non-`None`.

Backspin transition (`B`, `action_backspin`): the third transition mode alongside **SKIP** and **STRETCH**, but it bypasses `plan_transition()` entirely ‚Äî no rubberband, no crossfade. It abruptly stops the outgoing track, plays a backspin/rewind SFX one-shot, then starts the queued next track at its natural tempo from its **raw** cue (`panel.raw_cue` ‚Äî `0:00` or the exact typed value, NOT the bar-snapped cue the crossfade uses). Mechanically it builds a single buffer `np.concatenate([_backspin_audio, next_audio[cue_sample:]])` (where `cue_sample = int(raw_cue * SAMPLE_RATE)`) and hands it to `engine.play()` (played verbatim from index 0, like any other engine buffer). The SFX is decoded once at mount by `_preload_backspin()` into `self._backspin_audio` (a daemon thread, so the keypress has no decode hitch). By design it requires a **raw** next track: `action_backspin` rejects the press unless a track is playing, a next track is queued, `not self._preparing`, `self._next_prepared is None`, and the engine is not mid-transition (`state != MIXING` and `_t_restore_start == 0.0`) ‚Äî a prepared/stretched buffer is irrelevant to the SFX-then-play mechanic. The worker (`load_audio` + concatenate) hands off to `_apply_backspin()` on the UI thread, which plays the buffer, promotes the next track to now-playing, arms NO restoration ramp (natural tempo, like a SKIP), clears the NEXT slot, and offsets `_now_downbeats` by `len(_backspin_audio) - cue_sample` (dropping pre-cue downbeats) so a subsequent bar-aligned mix still lines up. Because the engine was already PLAYING and stays PLAYING, `_tick`'s MIXING‚ÜíPLAYING / ‚ÜíIDLE detectors don't fire ‚Äî no spurious ramp or `_handle_track_finished`.

Pre-listen cue / PFL (`L`, `action_cue_toggle`): `AutoMixApp` owns one optional `CuePlayer` alongside its `AudioEngine`, opened at mount by `_open_cue_device()` when `--headphones-device` resolved (`None` and dormant otherwise; a failed open is non-fatal). `L` toggles auditioning the queued NEXT track in the cue headphones: stopped ‚Üí bump `_cue_epoch`, set `_cue_loading`, and a daemon worker decodes `load_audio(next_path)[raw_cue_sample:]` (the **raw** cue, like backspin) and hands off to `_start_cue_playback`‚Üí`cue.play()`; playing-or-loading ‚Üí `_stop_cue()`. It is **independent of the master state machine** ‚Äî no `engine.state` guards, since cueing the next track over the current mix is the whole point. `_cue_epoch` mirrors `_prep_epoch`: bumped on every cue start/stop and by `_stop_cue`, so a decode that lands after the slot changed is discarded (both in the worker and a final check in `_start_cue_playback`). `_cue_loading` makes a second `L` during the decode read as stop (not a second decode). Lifecycle (Q7): the cue auto-stops via `_stop_cue()` when the NEXT slot is consumed/changed ‚Äî `action_mix_now`, `action_backspin` (in `_apply_backspin`), and `action_load_next` (only on an actual path change, after its no-op early-return); master transport (`S`/`Space`) and `C/F/R` never touch it. `[`/`]` seek ‚àì`CUE_SEEK_SECONDS` (5 s) via `cue.seek()`, handled in `on_key` **matched on `event.character`** (AltGr-safe) only while the cue is playing. `_tick` feeds `NextTrackPanel.set_cue_state(playing, pos, dur)` for the live `‚ô™ CUE  pos / dur  [bar]` line (rendered via `_progress_segment`) and surfaces a lost device once via `_cue_dead_reported`.

## Key invariants

- Navigation contract: the folder browser is a three-level drill-down ‚Äî **root display** (root node collapsed, song panel empty; this is the startup state, set by `_build_tree` which leaves the root collapsed and `on_mount` which no longer pre-loads songs) ‚Üí **subfolder display** (root expanded, subfolders visible) ‚Üí **song browsing** (a subfolder expanded + its songs loaded in the DataTable + focus on the DataTable). The three activation gestures ‚Äî right arrow, Enter, mouse click on a node's label ‚Äî drive each step: on the **root** node they only expand it (reveal subfolders, move the highlight onto the first subfolder, focus stays on the Tree); on a **subfolder** node they expand it AND load its songs AND move focus to the DataTable (right arrow via the `FolderTree.GoToSongs` message; Enter/click via `Tree.NodeSelected` ‚Üí `on_tree_node_selected`). Left arrow walks back up one level: in the DataTable it clears the song list, collapses the browsed subfolder, and returns focus to the Tree; in the Tree (`FolderTree.on_key`) it collapses the cursor node's **parent** and moves the highlight onto it ‚Äî so a single press from a first-level subfolder collapses the root and lands the highlight back on it (subfolder display ‚Üí root display) in one step. The root is never re-highlighted by up/down navigation: `on_key` intercepts `up` and blocks the step that would land on the root (`get_node_at_line(cursor_line-1) is root`), so once subfolders are shown the arrows only walk among them. `FolderTree` sets `auto_expand = False` so Enter/click never auto-toggle a node ‚Äî expansion is driven only by the handlers above; `on_key` also intercepts `space` (prevent_default to block Tree's built-in toggle) and forwards it to the global `action_toggle_pause` so spacebar never flips a folder's arrow while browsing. Because root activation never loads songs, songs living directly in the root folder (not in a subfolder) are not browsable by design. Do not add app-level left/right bindings ‚Äî they would shadow widget-level handling.
- All mutations to `AudioEngine` state (`_now_audio`, `_next_audio`, `_position`, `_mix_pos`, `state`) must be done inside `self._lock`. The callback also acquires the lock, so keep critical sections short.
- `_next_audio` passed to `start_mix()` must already be rendered by `make_transition_buffer()` (or be the full unmodified track for a non-tempo-matched cue) ‚Äî the engine plays it verbatim from index 0.
- After `MIXING ‚Üí PLAYING` transition, `_position` is set to `_mix_pos` (not 0), so playback of the former next track continues from where the crossfade left off.
- `call_from_thread` is the only safe way to touch Textual widgets from the audio/prep worker threads.
- The inline cue/fade text input (`_input_mode`) intercepts keys in `on_key` and calls `event.prevent_default()` + `event.stop()` to suppress bindings while active.
- `make_transition_buffer()` is the **only** place in the project that invokes rubberband. All time-stretching routes through it. The librosa import in `audio_engine.py` is gone; librosa is now used only by `analyzer.py` for BPM/key detection.
- Any change to Cue / Fade / Restore after Prepare must call `_mark_stale()`. The precomputed buffer's contents depend on all three; silently keeping a stale buffer was a pre-rubberband bug and must not return.
- The stretch rate is **always** `plan.start_rate` (the octave-folded `r_eff`), never the raw `now_bpm / next_bpm`. Computing it raw reintroduces the half-/double-time bug (a 64 BPM track stretched to 128). All callers go through `plan_transition()`; do not recompute the ratio by hand. `_next_plan` is held in lockstep with `_next_prepared` ‚Äî reset BOTH to `None` together everywhere (`action_stop`, `_handle_track_finished`, `action_load_next`, `_mark_stale`, and after consumption in `action_mix_now`). A `_next_plan.skip == True` buffer is the **raw** cue audio (no rubberband); `action_mix_now` must arm no restoration ramp for it (`_restore_from_bpm`/`_restore_to_bpm` left at `0.0`), because a skipped track plays at its natural tempo with nothing to restore. `MAX_DRIFT_BEATS` (0.25) in `transition.py` is the single tunable that defines "tempos close enough to skip"; the implied BPM window scales inversely with fade length (`|Œîbpm| ‚â§ 60¬∑MAX_DRIFT_BEATS/fade`).
- `action_mix_now()` is blocked while `engine.state == MIXING` or `_t_restore_start > 0.0` (i.e. previous transition's crossfade or restoration ramp still running). The prepared buffer's `start_rate` is rendered against `_now_bpm` (the natural BPM of what's playing), which only equals the *actual* playback BPM once any prior ramp has finished. Preparing during a transition is fine; mixing into one is not. The block messages are phase-specific ("Mixing the two tracks ‚Äî cannot mix another track yet." vs "Restoring original tempo ‚Äî cannot mix another track yet.") and match the `NowPlayingPanel._phase` banner that `_tick()` keeps in sync with the same two conditions. `_tick()` sets the banner to one of four states whenever a track is playing (`state != IDLE`): "MIXING the two tracks - cannot mix another track" during the crossfade, "WAITING for downbeat - cannot mix another track" while `_mix_scheduled` (a bar-aligned mix is armed but the trigger downbeat hasn't arrived yet), "RESTORING original tempo - cannot mix another track" while `_t_restore_start > 0.0`, and otherwise "Ready to mix another song" ‚Äî i.e. the idle-of-transition state where `action_mix_now` would be accepted. During the crossfade only, the panel title shows both tracks as "outgoing ‚Üí incoming": `_apply_now_swap` passes the outgoing name to `NowPlayingPanel.set_track(mix_from=...)` (captured before `_now_path` is overwritten), and `_tick` calls `panel.clear_mix_from()` at MIXING‚ÜíPLAYING so the dual name reverts to the single incoming track once the crossfade ends (`clear()`/`set_track` without `mix_from` also reset it, covering Stop / track-end / load paths). The progress line mirrors this: during MIXING, `_tick` passes the engine's `mix_position`/`mix_duration` (the incoming buffer's playhead/length ‚Äî `_mix_pos` and `len(_next_audio)`, exposed as engine properties) to `refresh_progress`, which renders two 18-wide bars "outgoing ‚Üí incoming" side by side via `_progress_segment()`; outside MIXING it passes `None`/`None` and falls back to the single 32-wide bar. Note the incoming buffer is the whole remaining track, so its bar sits near-empty during the short fade window (accurate, not a bug). Both bars MUST build their `[...]` through `_progress_segment`, whose leading bracket is escaped `\[` ‚Äî an unescaped `[#...` is parsed by Rich as a colour tag and the bar silently vanishes (see the Rich-markup gotcha).
- Now-playing track changes ONLY through `Enter` (when engine IDLE) or through a completed Mix; pressing `Enter` while anything is playing is rejected by `on_data_table_row_selected`. This means `_do_load_now` is never called against a non-IDLE engine via the UI ‚Äî keep that invariant when wiring any new callers.
- `action_prepare_mix` and `action_mix_now` both require the engine to be actually playing (`state != IDLE`, and for Prepare `_now_bpm > 0`). Prepare's plan rate `r_eff` (from `now_bpm / next_bpm`) is degenerate with nothing playing; Mix's `engine.start_mix` no-ops on a non-PLAYING engine, so without the guard the app would desync (clear the NEXT panel and claim "Mixing‚Ä¶" while the engine produced silence). Keep both guards if you add new entry points.
- When the engine reaches IDLE on its own (track played to the end, not an explicit Stop), `_tick()` must call `_handle_track_finished()` exactly once ‚Äî guarded by `_now_path is not None` so it doesn't re-fire after `action_stop` (which already nulls `_now_path`). The handler invalidates prep via `_prep_epoch` because any prepared buffer's `start_rate` was rendered against the track that just ended.
- `_do_load_now()` (Enter on a song row) must clear `_t_restore_start`, `_restore_from_bpm`, `_restore_to_bpm`, `_mix_scheduled`, AND bump `_prep_epoch` (plus clear `_preparing`/`_prep_animating`/`_prep_progress`) before spawning its load worker. Otherwise a prior transition's BPM-display ramp continues animating against the freshly loaded track, and ‚Äî if Enter interrupted an active crossfade ‚Äî the next `_tick()` would re-arm the ramp via the MIXING‚ÜíPLAYING detector. Also, an in-flight Prepare worker was rendering against the OLD `_now_bpm`; without the epoch bump it would install a wrong-tempo buffer when it completes. The engine's `play()` only swaps `_now_audio`; it doesn't know anything about app-level restoration or prep state.
- `_prep_epoch` is the single source of truth for "is the current prep worker still relevant?". Bump it (and ideally clear `_preparing` / `_prep_animating` / `_prep_progress`) in EVERY action that invalidates an in-flight prep: `action_load_next` (queued next-track changed), `action_prepare_mix` (new prep supersedes any old one), `_mark_stale` (C/F/R changed ‚Üí old buffer's parameters wrong), `action_stop`, `_do_load_now` (now-playing changed ‚Üí `r_eff` denominator wrong), and `_handle_track_finished` (track ended ‚Üí buffer matched a track no longer playing). Each worker captures the epoch at launch and aborts on completion if it differs. NOTE the epoch only orphans *in-flight* workers ‚Äî actions that can leave a **completed** `_next_prepared` buffer dangling against a no-longer-current now-playing track (`action_stop`, `_handle_track_finished`) must ALSO set `_next_prepared = None` (and `_next_plan = None`) and reset the NEXT panel to `[not prepared]`, or a later `Enter`‚Üí`M` would mix a wrong-tempo buffer.
- `NextTrackPanel` keeps **two** cue values in seconds, both written only by `set_cue()`: `_cue_raw` is the unsnapped value (`0.0` or exactly what the user typed; exposed as `raw_cue`), and `_cue` is the bar-snapped version (nearest downbeat when `_downbeats` is non-empty, else identical to raw; exposed as `cue`, flag `cue_snapped`). **Backspin (`action_backspin`) reads `raw_cue`; Prepare/Mix (`action_prepare_mix`, `action_mix_now`) read the snapped `cue`** ‚Äî so a `0:00` cue backspins from the track start while the crossfade still bar-aligns. Any new code that writes `_cue`/`_cue_raw` directly (bypassing `set_cue`) breaks this split and the bar alignment. The panel renders both as `Cue: <raw>   Mix: <snapped> [bar]` (the `Mix:` field only when snapped). The `cue_snapped` flag is the source of truth `action_mix_now` uses to decide whether bar-aligned scheduling is applicable.
- `action_mix_now` only schedules a deferred (bar-aligned) crossfade when BOTH tracks have downbeats AND the cue was snapped. With a half-aligned pair (only one side has downbeats), scheduling would delay the mix without making it any more musical, so it falls back to immediate start. Status line reports which path was taken.
- The backspin transition (`action_backspin`, `B`) requires a **raw** next track and bypasses the prepare pipeline. Its guards ‚Äî track playing, `_next_path` set, `not _preparing`, `_next_prepared is None`, `state != MIXING`, `_t_restore_start == 0.0`, `_backspin_audio` loaded ‚Äî must all hold; a prepared OR being-prepared next track is explicitly rejected (the SFX-then-play mechanic never uses the prepared buffer). It starts the next track from the **raw** cue (`panel.raw_cue`, NOT the snapped `cue`), arms no restoration ramp (leave `_restore_from_bpm`/`_restore_to_bpm` at `0.0`, like a SKIP), and MUST offset `_now_downbeats` by `len(_backspin_audio) - cue_sample` (prepended SFX + raw-cue cut), dropping pre-cue downbeats, or a later bar-aligned mix lands off the bar. It calls `engine.play()` (not `start_mix`), so the engine stays PLAYING and `_tick` arms nothing ‚Äî do not route it through the MIXING transition detectors.
- The library dict that the app and analyzer share has the shape `{path: {bpm, key, beats, downbeats}}` (a per-track record dict, NOT a `(bpm, key)` tuple). Use `empty_record()` from `analyzer.py` as the default in `library.get(path, ...)` calls; missing/garbled cache entries become an empty record rather than a KeyError.
- The pre-listen cue (`CuePlayer`, `action_cue_toggle`, `L`) is a **second, independent output stream** that must never couple to the master `AudioEngine`: separate stream/lock/buffer, no shared state, and no `engine.state` guard (cueing is allowed in any engine state ‚Äî that's the point). Its decode reads the **raw** cue (`panel.raw_cue`), never `_next_prepared` (a STRETCH buffer is the wrong tempo for a natural-tempo preview, and reusing it would re-introduce the prep coupling the separate class exists to avoid). `_cue_epoch` and `_cue_loading` are held in lockstep the way `_prep_epoch` is: bump `_cue_epoch` in `_stop_cue` AND at every cue start; a worker captures it and discards a stale buffer in BOTH the worker and `_start_cue_playback`. `_stop_cue()` is the single auto-stop path ‚Äî call it (not `cue.stop()` directly) from `action_mix_now`, `_apply_backspin`, and `action_load_next` (on an actual NEXT-slot change), so the epoch bump and `_cue_loading` reset happen together. Master transport (`S`/`Space`) and `C/F/R` must NOT stop or reposition a live cue. A `CuePlayer` exception (device lost mid-set) must be swallowed inside `CuePlayer` and surface only as `is_dead` ‚Üí a one-shot status line via `_cue_dead_reported`; it must never reach the master engine or crash the UI. `[`/`]` seek MUST match `event.character` (not `event.key`) so AltGr-composed brackets work on non-US keyboard layouts (e.g. Italian); the volume keys (`,`/`.` master, `9`/`0` cue) match `event.character` for the same reason.
- Software volume lives in the audio layer, not the app: `AudioEngine._volume` (clamped `[0.0, MAX_GAIN]`, boost-capable, output hard-limited to ¬±1.0 when >1.0) and `CuePlayer._volume` (clamped `[0.0, 1.0]`, no boost ‚Äî ear safety) are applied inside their callbacks. The app only adjusts them via the `,`/`.` (master) and `9`/`0` (cue) keys and reads `engine.volume` / `cue.volume` in `_tick` to render `Vol NN%` / `Vol NN% (boost)` (NowPlaying) and `vol NN%` (the `‚ô™ CUE` line). The master volume is independent of the OS default-device volume keys, which is the whole point ‚Äî pinning `--main-device` to a non-default output otherwise loses hardware volume control.

## Gotchas

**Textual version ceiling** ‚Äî `requirements.txt` pins `textual<1.0.0`. Textual 1.x+ (now at 8.x) rewrote internal rendering APIs (`Visual`, `Content`, `render_strips`) in ways that break `Static` subclasses. Do not bump past `0.x`.

**Rich markup in `Static.update()`** ‚Äî Textual passes strings through Rich markup parsing. Any literal `[`, `]` in displayed text (e.g. `[none]`, `[C]`, filenames) must be escaped as `\\[`. Use `rich.markup.escape()` for user-supplied strings (filenames). Failure produces `AttributeError: 'NoneType' object has no attribute 'render_strips'` at render time.

**WSL audio** ‚Äî `sounddevice` has no audio devices under WSL without WSLg (WSL2 + `wsl --update`) or manual PulseAudio-over-TCP setup. Run natively on Windows or a Linux desktop instead.

**Console output must be ASCII** ‚Äî Windows terminals default to cp1252. The `main.py` progress bar uses `#`/`-` (not `‚ñà‚ñë`), and *every* string printed outside the Textual TUI ‚Äî argparse `help=`, `sys.stderr` errors, status text ‚Äî must avoid em-dashes/Unicode (`‚Äî` ‚Üí `-`), or you get `ÔøΩ` mojibake or `UnicodeEncodeError` before the TUI starts. (Rich-rendered text inside Textual is fine.)

**MME truncates device names to 31 chars** ‚Äî `sd.query_devices()` lists each output under several host APIs; the MME copy's name is cut to 31 chars, so a long `--main-device`/`--headphones-device` substring can skip the MME entry and match the DirectSound/WASAPI copy (same physical device, different index). `_resolve_output_device` returns the first output-capable match ‚Äî prefer a short substring (`Realtek`, `B01`); `--list-devices` shows the indices.

**Textual key event order** ‚Äî Within a widget, `on_key` fires *before* the widget's own BINDINGS dispatch; `event.prevent_default()` + `event.stop()` in a subclass `on_key` successfully blocks built-in key actions. App-level `on_key` fires *after* the focused widget has already acted, so intercepting a key before Tree or DataTable handles it requires subclassing the widget, not handling it in the App. Also: `Tree.NodeSelected` fires only on Enter/click, not on arrow-key navigation; use `Tree.NodeHighlighted` for cursor-movement events.

**Rubberband stderr parsing** ‚Äî Rubberband uses **carriage-return overwrite** for its `<n>%` progress digits (not newlines), so `readline()` and the default line-buffered iterator both block until the pass ends, defeating real-time progress. `_drain_stderr_with_progress` reads stderr one character at a time and chunks on either `\r` or `\n`. Also: in R3 mode Pass 1 ("Studying") is essentially instant (~1% of wall time) and Pass 2 ("Processing") does ~99% ‚Äî a naive 50/50 weighting causes the BPM display to snap to halfway in the first frame and crawl the remainder over many seconds. The constants `_PASS1_WEIGHT = 0.05` / `_PASS2_WEIGHT = 0.95` in `stretcher.py` reflect this asymmetry; if you ever switch to the R2 engine these need re-calibrating.

**Rubberband CLI flag trap** ‚Äî `-T` is `--tempo` (a constant stretch factor), NOT the timemap flag. The timemap flag is `-M` (or `--timemap`), and when used it requires an overall stretch factor alongside it, supplied via `-D <duration_seconds>` (or `-t`/`-T`). Without `-D`, rubberband silently produces input-length output with no stretching applied. Also: an explicit `(0, 0)` first row in the timemap triggers a NaN-time-ratio computation at sample 0 that corrupts output; rubberband anchors `(0, 0)` implicitly, so the first explicit anchor must be strictly past origin (handled by `_build_timemap`'s monotonic filter).

**Downbeat detection is heuristic** ‚Äî `analyzer.py` runs librosa's `beat_track` then assigns downbeats by picking the phase (`0,1,2,3`) that maximises summed 40‚Äì150 Hz STFT energy at every 4th beat. Works well on 4/4 dance/pop (consistent kick on the 1); less reliable on jazz, 3/4 material, or tracks with no clear kick drum. A track with no detected beats (atonal/very short) gets empty `beats`/`downbeats` lists, and any mix involving it falls back to immediate (non-aligned) start ‚Äî `action_mix_now` makes this decision explicitly.

**madmom on Python 3.12 is impractical** ‚Äî Tried madmom (more accurate downbeats) and BeatNet (which also needs madmom). Madmom's PyPI release is from 2018, uses `np.float` (removed in numpy 1.24), and requires MSVC C++ build tools to compile. Going that route demands pinning `numpy<2` + installing Build Tools. The librosa heuristic is the conscious trade-off for keeping the deps simple.

**Cache schema versioning** ‚Äî The cache file at `~/.automix_cache.json` is now `{"version": 2, "entries": {...}}`. The loader rejects any other shape and re-analyzes from scratch ‚Äî so upgrading the schema is automatic but the first launch after a bump pays the full re-analysis cost. Bump `CACHE_VERSION` whenever the per-track record's required fields change.

## Audio data format

All audio arrays are `(N, 2) float32` in `[-1.0, 1.0]`, at 44100 Hz. pydub decodes to 16-bit PCM integers (`/ 32768.0` normalises). Analysis uses a separate 22050 Hz mono path inside `analyzer.py` and never shares arrays with the engine.

**Headless harness pattern** ‚Äî the engine and Textual UI are tested without an audio device or a mounted app: build the `AudioEngine` via `AudioEngine.__new__(...)` and set its state fields manually (skips opening the `sd.OutputStream`), then drive `_callback`/`_fill_playing` directly with synthetic numpy audio. For app-level logic, build `AutoMixApp.__new__(...)`, stub `query_one` to return fake panels and `_status` to record messages, and call `_tick()` / action methods directly. Keep this pattern when adding tests ‚Äî a real stream or mount can't run in CI/headless. When you add a field to `AudioEngine`/`CuePlayer.__init__` or new `AutoMixApp` state, update every `__new__` harness that builds the object (and the `FakeEngine`/`FakeCue` stand-ins) to set it too ‚Äî the harnesses bypass `__init__`, so a missing field surfaces as an `AttributeError` from the driven `_callback`/`_tick`/action, not a clean skip.

`tests/diagnose_progress.py` is not a test ‚Äî it's a diagnostic that runs `make_transition_buffer` on a real track and logs every rubberband stderr chunk + parsed progress to `tests/progress_log.txt` (used to debug the prep-progress oscillation bug).
